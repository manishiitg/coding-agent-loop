## Schedule Management

You can create scheduled tasks that run automatically on a cron schedule. Multi-agent (Chief-of-Staff) schedules are stored in `_users/<user-id>/multiagent-schedules.json`, but you manage them **only through the dedicated tools below** — the server writes that file for you.

> **Never edit `multiagent-schedules.json` directly** (no `execute_shell_command`, `python3`, `jq`, or file writes). That path is protected by the workspace folder guard, and direct edits can silently drop built-in/system entries (e.g. a memory-enrich schedule) or corrupt the file. Always use the tools — they read-modify-write the whole file server-side, preserving entries you are not changing, and immediately (re)activate the schedule in the running scheduler.

> Scope: this doc covers **multi-agent chat** schedules only, which are **cron-only** (a repeating cadence). To schedule a **workflow** — including dated, one-time **calendar** schedules (a fixed list of specific date/time runs, e.g. a content calendar) — use the workflow-schedule tools (`create_workflow_schedule` / `create_calendar_workflow_schedule`) documented in `get_reference_doc(kind="workflow-tools")`; those live in the workflow's `workflow.json`, not here.

### Tools

- `list_multiagent_schedules` — list this user's schedules. Each entry shows the `id`, name, cron, timezone, enabled state, next run (UTC), and query. **Call this first** to see what already exists and to get the `id` other tools need. Pass `enabled_only: true` to hide disabled entries.
- `create_multiagent_schedule` — create a new cron schedule. Required: `name`, `cron_expression`, `timezone`, `query`. Optional: `description`, `enabled` (defaults true). The server generates the `id`, validates the cron and timezone, writes the file, and activates the schedule.
- `update_multiagent_schedule` — patch an existing schedule by `schedule_id`. Only the fields you pass change; omitted fields keep their current values. To **pause** a schedule set `enabled: false` (this preserves the rest of the definition); to resume set `enabled: true`. `query` cannot be set to empty.
- `delete_multiagent_schedule` — permanently remove a schedule by `schedule_id`. Cannot be undone.
- `trigger_multiagent_schedule` — run a schedule immediately by `schedule_id`, outside its normal cron timing.
- `get_multiagent_schedule_runs` — recent execution history for a schedule (`schedule_id`, optional `limit`, default 10): status, duration, session id, errors.

### Entry fields

Each schedule the tools create/maintain has:

- `id` — generated by the server (a UUID). You never set this.
- `name` — human-readable display name.
- `description` — optional note about what the schedule does.
- `cron_expression` — 5-field cron (minute hour day-of-month month day-of-week).
- `timezone` — required IANA timezone (e.g. `UTC`, `America/New_York`, `Asia/Kolkata`). Do not use abbreviations like EST/PST/IST.
- `enabled` — whether the schedule is active.
- `mode` — always `multi-agent` for these schedules (set automatically).
- `query` — the message/instruction executed on each scheduled run.

### Cron expression examples

- `0 9 * * *` — daily at 9:00 AM
- `0 9 * * 1-5` — weekdays at 9:00 AM
- `*/30 * * * *` — every 30 minutes
- `0 0 1 * *` — first day of each month at midnight

### When users ask to schedule something

1. Confirm the schedule details (what to run, when, timezone).
2. Call `list_multiagent_schedules` to check for an existing/overlapping schedule.
3. Call `create_multiagent_schedule` with `name`, `cron_expression`, `timezone`, and the user's instruction as `query`.
4. Confirm to the user what was scheduled (include the returned `id` and next run time).

### Updating, pausing, or removing schedules

- To **update**: call `update_multiagent_schedule` with the `schedule_id` and just the fields to change.
- To **pause temporarily**: `update_multiagent_schedule` with `enabled: false` (keeps the definition intact); resume with `enabled: true`.
- To **remove**: `delete_multiagent_schedule` with the `schedule_id`.
- To **run now**: `trigger_multiagent_schedule` with the `schedule_id`.

The scheduler activates tool changes immediately — no manual file reload is needed.

### Multiple-user note

These tools always operate on the calling user's own schedule store; you do not need to know or pass the user id. If no multi-tenant identity is set, the store defaults to the `default` user.
