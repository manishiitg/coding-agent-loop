How to use this PRD

Product promise

A supervised AI teacher that learns how each child is progressing, teaches through attempts instead of simply revealing answers, and gives the parent an evidence-based view of learning.

Design source of truth

Every trusted React screen and agent-generated HTML learning view must follow family-learning-design-guidelines.md. That file defines the SparkQuill Sunlit Canvas tokens, typography, shared Parent/Child chat shell, setup-screen consistency, asset visibility rules, accessibility baseline, and forbidden patterns.

The images in family-learning-assets/ are visual explorations that help us reason about structure. Some still show an earlier working label; treat them only as layout and interaction references until branded screens are regenerated. They do not override the written guidelines or become implementation contracts until a direction is explicitly selected. Impeccable is installed locally at .github/skills/impeccable/ for layout, typography, and polish checks; it refines the shared guidelines rather than replacing them.

Runtime source of truth · AgentWorks chief of staff & automations are the gold standard

SparkQuill is not a new runtime. Its agent layer replicates and reuses the two proven AgentWorks patterns — the chief-of-staff chat and automations / workflows — from mcp-agent-builder-go and the shared mcpagent / multi-llm-provider-go libraries. Where SparkQuill diverges from how AgentWorks does it, that is a bug to fix, not a new design. Copy the pattern before inventing one.

Chief-of-staff chat → the Parent Learning chat (and, with the same runtime, the Child tutor and parent WhatsApp threads). AgentWorks runs a coding agent (Claude Code / Codex / Cursor) through the mcpagent MCP bridge with an in-process executor server, and drives it turn after turn in one conversation. SparkQuill's parent chat is the same chief of staff, scoped to one family: the company metaphor holds — the parent is CEO, the chat is the chief of staff, skills are the employees, and each skill is one capability.

Automations / workflows → SparkQuill skills. Every unit of parent-facing work — process a file, create a test, build study material, write a progress report, back up, publish, notify — is an agentic, file-system-based skill the chat invokes through the same bridge, exactly as an AgentWorks workflow runs one capability. There are no hand-coded JSON CRUD endpoints for this work; the agent does it with tools + skills against the filesystem.

Session & bridge lifecycle must follow AgentWorks, which means:

  • One process-global executor / MCP bridge, started once at server boot on a fixed port and shared by every conversation and skill run — not a fresh bridge per HTTP request. (AgentWorks: the bridge is the main server's own route set, wired once at startup; the persistent CLIs call back into that always-alive port.) Standing up a new bridge per request leaves the persistent CLI calling a dead port on the next turn — the exact class of bug this rule prevents.
  • The warm coding-agent (tmux) session is reused across turns via a session registry keyed by conversation / session id — never cold-started per turn. The lightweight per-turn agent wrapper may be rebuilt cheaply, but the bridge and the persistent tmux CLI must survive between turns; that is what makes resume warm instead of a cold restart with history replay.
  • Cleanup is by per-session idle timeout plus a periodic tmux reaper — the way AgentWorks arms a per-session idle timer (default ~3h, reset on each turn) and runs a reaper that force-kills orphaned coding-agent tmux process trees — not by an LRU or size cap. Do not add bespoke eviction machinery AgentWorks does not have.

File-first storage still governs data (filesystem for material, conversations, attempts, learning documents, generated HTML; SQLite/state for identity, permissions, and indexes). Server-owned policies — identity, the child sandbox, answer-key isolation, notification recipients — remain the enforcement boundary regardless of what the agent proposes.

Working name decision · SparkQuill

SparkQuill is the working customer-facing name. It feels playful enough for children, credible enough for parents, and carries a subtle magical-school quality without directly borrowing a franchise identity. The learning guide may be called Quill, and the working tagline is Learning that grows with you.

Why this name:

  • Spark represents the moment a child begins to understand something, curiosity being awakened, and the confidence to try.
  • Quill connects naturally to writing, schoolwork, reflection, and a patient guide that learns alongside the child. It also gives the in-product tutor a friendly name: Quill.
  • Together, the name supports the product promise: turn small moments of understanding into a learning record that becomes more personal over time.
  • For children, it feels imaginative and lightly magical without sounding babyish. For parents, it still feels educational, trustworthy, and suitable for both a third-grader and a teenager.
  • Brand boundary: use the broad feeling of discovery, books, and a magical school—not names, symbols, characters, language, or visual motifs taken from Harry Potter or another franchise.

This is a product-name decision, not legal clearance. Trademark, domain, app-store, and company-name availability must be checked before public launch.

Enter the Parent Learning chat

This chat is the centre of Parent Mode. The first subject and topic—and later progress, materials, and tests—are handled here rather than through separate setup or dashboard screens.

SScreen

This is the first real chat with the learning guide. It follows the child profile and asks the parent what the child is working on now. The parent can answer naturally—for example, “Maya has a Mathematics test on quadratic equations next week”—or attach a worksheet, textbook page, or assignment.

There are no Subject and Topic input fields, no complete subject list, no syllabus setup, and no academic interview.

If an important detail is missing, the agent asks one focused follow-up question. Otherwise it accepts an incomplete starting point and lets the parent continue immediately.

MVP direction: after the two setup screens, the product becomes a full-height three-column operating shell. A narrow left rail holds Back, the child, new conversation, and recent sessions; the flexible centre conversation remains visually dominant; and a collapsible right drawer holds current context, sources, generated assets, review items, and the child handoff. There is no separate full-width page header or large hero heading. The same shell is reused in Parent and Child modes with different permissions, prompts, assets, and tone.

What parents do inside the chat

Current chat-shell design direction

The left rail is not feature navigation, the right drawer is not a dashboard, and neither mode has a separate full-width header. Selection between these visual directions remains open.

1 · Understand progress

Explain recent work, strengths, sticking points, missing work, and the next useful focus. Link important claims to conversations, attempts, or materials.

2 · Suggest a plan

Recommend a small next plan: continue a topic, revise, practise, upload a missing worksheet, or prepare for a test. The parent chooses what to use.

3 · Create study material

Create a child-ready explanation, revision sheet, practice set, flashcards, or worked-example guide based on the active subject, topic, and permitted material.

4 · Create a test

Create a short quiz, worksheet, or timed practice test with an age-appropriate difficulty and clear goal. Keep answers and marking guidance parent-private.

5 · Tell the guide

Tell the agent what changed, attach school material if useful, or ask it to update the current focus. The agent proposes the smallest safe update.

6 · Review before sharing

Preview a generated material or test, then either approve it for the child or tell the agent what to change. Do not expose manual editors, delete controls, or complex configuration screens.

Onboarding and voice direction

Onboarding outcome

A parent should move from first launch to one child’s first useful learning conversation without configuring the whole product. Ask only for information needed now; teach advanced features after the first successful session.

Voice direction: preferred later, optional always

The long-term child experience should be voice-centric because speaking is often easier and more natural than typing, especially for younger children. Text must remain fully supported for privacy, accessibility, accuracy, quiet environments, equations, and fallback. The first MVP will not implement audio; it will keep the action/event architecture ready for voice and introduce an optional local/open-source voice add-on later.

What the onboarding should feel like

  • Parent-led: onboarding always begins in protected parent setup. The child is invited only after permissions and the first profile exist.
  • One decision per screen: large title, short explanation, one primary action, a visible Back action, and Skip only for genuinely optional items.
  • Calm and local: explain that learning data remains in the separate SparkQuill workspace and that the installed agent runs locally.
  • Progressive: create one child and one learning path first. Additional children, subjects, WhatsApp, schedules, voice, and advanced preferences come later.
  • No empty-dashboard ending: onboarding ends by starting a sample conversation or adding the child’s first real school item.
  • Sunlit visual language: navy foundation, yellow primary actions, coral attention states, rounded white cards, generous spacing, and friendly plain language.

Core experience

Information structure

Family → Child → Grade → Subject → Chapter or topic → Assignment → Learning session

Supported material

Parents and children can upload PDFs, school notes, textbook pages, worksheets, notebook photographs, homework, answer keys, marking schemes, exam schedules, and teacher feedback. The system proposes where each item belongs and asks for confirmation when uncertain.

Parent journey

Add or forward material → choose the child and subject → set an assignment or goal → configure help level → monitor progress → review the summary and next step.

Child journey

Open today’s plan → learn or practise → attempt the work → request a hint → submit an answer or photograph → receive feedback → continue.

Grounding rule

When teaching or grading from supplied school material, the AI should identify the source document and page whenever possible. It must distinguish source-based information from general knowledge.

Decision: curriculum- and language-agnostic

The product should not be limited to a predefined list of curricula or languages. It should aim to support any curriculum and language the connected AI model can handle, while learning the child’s exact syllabus from uploaded textbooks, school notes, assignments, marking schemes, and parent instructions. Broad model knowledge is useful, but supplied school material remains the authority when accuracy matters.

06

Learning agents

Several specialized prompts or agents can work behind the interface while appearing to the family as one consistent teacher.

Tutor

Explains concepts, checks understanding, and tries a different explanation when the first one does not work.

Socratic coach

Uses questions and graduated hints instead of immediately revealing final answers.

Learning planner

Builds a manageable plan using assignments, deadlines, learning gaps, and available study time.

Careful grader

Reads submitted work conservatively, applies a rubric, shows evidence, and escalates unreadable work.

Quiz maker

Creates age-appropriate practice from the child’s own school material.

Progress analyst

Summarizes mastery, effort, misconceptions, and recommended next actions for the parent.

Student model

Maintain a parent-reviewable record of concept mastery, recent attempts, recurring misconceptions, hint usage, preferred explanation style, pace, confidence, interests, last-practised date, and scheduled follow-up. Use this history to personalize what is taught next, how it is explained, how difficult the practice should be, and when a topic should be revisited. Do not make hidden psychological judgments about the child.

Personalization goal

The experience should become meaningfully more personal over time. Two children in the same grade studying the same chapter may receive different examples, step sizes, practice difficulty, revision schedules, and teaching approaches based on what the system has learned from their actual progress.

07

Careful grading protocol

The AI must behave like a careful teacher, not an agreeable chatbot. It must never guess an unclear handwritten expression and then mark its own guess as correct.

ResultMeaning
CorrectThe visible work supports the expected answer and method.
IncorrectThe work is readable and conflicts with the expected answer or method.

Evidence required for every grade

  • What the AI believes the child wrote
  • Confidence in that transcription
  • The marking rule or rubric used
  • Why marks were awarded or deducted
  • Whether parent review is required
Unreadable work

If a photograph is blurred, cropped, shadowed, incomplete, or ambiguous, ask for a clearer image or typed transcription. Do not reveal the answer simply because the submission could not be read.

Implementation guidelines

Recommended approach

Publish SparkQuill as a separate Electron desktop app, but build it in the same repository and on the same backend, agent runtime, frontend services, and component library as AgentWorks. Give each app its own frontend entry, shell, branding, application identity, installer, and release channel. Reuse platform capabilities and small UI primitives, but do not reuse the current coding-agent screens as the child or parent experience.

Confirmed technical foundation

Use the existing mcp-agent-builder-go repository as the shared foundation for AgentWorks and SparkQuill. Add the new learning frontend entry, education domain layer, and separately branded Electron build there instead of starting a disconnected repository.

Chat-centred design references

The learning experience is a conversation, not a dashboard with chat added later. These references define three compatible layouts. The first is the recommended MVP; the other two show interaction patterns that can be introduced without changing the underlying learning-session architecture.

UI direction

Build Reference A as the desktop shell, borrow Reference C’s inline learning-event cards, and defer Reference B’s editable working canvas until the chat, upload, grading, and personalization loop is proven.

Why this is the lowest-effort path

The current product already contains valuable infrastructure for authentication, agent providers, conversations, file uploads, document rendering, scheduling, WhatsApp, notifications, and desktop connectivity. Rebuilding those capabilities would be expensive. However, the current ChatArea, ChatInput, Workspace, and bot-configuration screens are large and designed for technical users. Trying to hide their complexity with conditionals would create a fragile frontend and a poor experience for children.

Desktop access model

Child mode · default

After setup, the app reopens the last-selected child for speed and provides a clearly visible profile switcher. On first use, or when no previous profile is available, it shows a child chooser. Children do not need a PIN for the MVP. They see only their learning plan, subjects, tutor sessions, practice, and submissions.

Parent mode · protected

A Parent Mode action requires a parent PIN. Parent mode contains setup, private notes, answer controls, grading review, progress, schedules, WhatsApp, agent configuration, and data management. A clear “Forgot PIN” flow lets the verified parent reset it without losing family learning data.

  • First launch is a parent setup flow: create the PIN, connect or detect Codex or Claude Code, and create child profiles.
  • Store the PIN securely using the operating system keychain or Electron secure storage; never store it in frontend local storage or plain text.
  • Allow easy PIN reset after parent verification through the signed-in parent account, an email code, operating-system authentication, or another approved recovery method. The exact verification method must be chosen before implementation.
  • Resetting the PIN must not delete child profiles, learning history, uploads, schedules, or parent settings.
  • Never expose a PIN reset path that a child can complete without parent verification.
  • Rate-limit incorrect attempts and automatically lock parent mode after inactivity or when the app returns to child mode.
  • Enforce the child restrictions in the backend as well as the frontend. Child sessions must not receive terminal, arbitrary file, model, provider, or unrestricted tool access.
  • The PIN provides practical separation on a family computer, not protection from a child with full access to the same operating-system account. Stronger security would require separate OS accounts or full parent authentication later.

Backend and agent boundaries

  • Add a thin education domain layer for children, subjects, materials, assignments, learning sessions, progress, and grading reviews.
  • Continue using the existing agent runtime underneath, but invoke it through education-specific actions rather than unrestricted general chat.
  • Store tutor, coach, grader, planner, and progress policies as versioned server-side prompts or workflows—not frontend text.
  • Return structured results for grading, including transcription, confidence, rubric evidence, marks, and review status.
  • Enforce parent and child permissions on the server. Hiding controls in the frontend is not sufficient.
  • Keep the provider layer interchangeable so a family can use Claude Code, Codex CLI, Cursor Agent, Pi, or future integrations without changing the learning UI.

One message from click to response

  1. The child chooses an intent: Ask, Explain, Give me a hint, Check my attempt, Start practice, or Submit work. The composer sends a typed action, not a hidden terminal command.
  2. The learning API validates access: it checks the current child, mode, subject, assignment, allowed action, answer-reveal policy, and rate limits.
  3. The context broker prepares a run package: it reads structured state from SQLite and includes only the permitted child profile summary, current material, prior attempts, relevant concept memory, and selected source files. Parent notes and answer keys are excluded unless the chosen workflow explicitly permits them.
  4. The session controller resumes or starts tmux: it maps the learning-session ID to one provider process running inside that session’s restricted filesystem directory. Provider selection remains a parent setting.
  5. The workflow sends a versioned instruction: the tutor, coach, quiz, grader, planner, or progress workflow receives the action and context package. The frontend does not assemble the system prompt.
  6. The bridge normalizes live output: provider-specific output becomes a small shared event format. Partial tutor text can stream immediately; tool activity stays hidden unless it produces a child-safe learning artifact.
  7. The server validates and stores: transcripts and artifacts are written to the filesystem, required structured fields are validated, and SQLite indexes/status are updated in a transaction.
  8. React renders the result: ordinary explanations appear as tutor messages; hints, questions, source evidence, uncertainty, grading feedback, and progress updates use distinct cards from Reference C.

Runtime rule

The tmux session name must be an opaque generated identifier with no child name or personal information. Its working directory, readable files, writable paths, environment variables, tools, and maximum run time must come from a server-owned child-safe profile. A child message is data for a tutor workflow—not permission to execute a shell command.

Make the MVP voice-ready without implementing audio

  • Keep learning actions modality-neutral: the backend receives text plus an input-modality field. Today the text is typed; later it may be a parent/child-confirmed speech transcript.
  • Keep output modality-neutral: the tutor produces validated text events. The UI renders them as captions and may later send eligible text to a local text-to-speech adapter.
  • Define adapters now, implement later: reserve interfaces for speech-to-text health/transcription and text-to-speech health/synthesis without adding audio libraries, permissions, installers, binaries, or UI to the MVP.
  • Do not run speech through tmux: Codex or Claude Code remains the reasoning teacher; the optional speech runtime is a separate local sidecar with its own lifecycle and health check.
  • Preserve turn control: a voice transcript must pass through the same child permissions, answer policy, context broker, event schema, persistence, and audit path as typed input.
  • Keep provider choice open: treat FluidVoice as a candidate connector and retain interchangeable speech adapters until the project, licence, operating-system support, performance, and quality are verified.

Learning action and event contract

A small typed contract keeps the new UI independent of Codex, Claude Code, tmux, and prompt wording. Exact URLs may follow the existing API conventions, but the concepts below should remain stable.

UI sends actions
  • ask child question with optional material references
  • request_hint next hint level, subject to reveal policy
  • explain concept or selected feedback in a chosen style
  • check_attempt formative feedback without final submission
  • submit_attempt freeze an attempt and start careful grading
  • answer_question response to a tutor or quiz question
  • confirm_intake parent approval of classification and destination
Server streams events
  • tutor_message safe Markdown segments with citations
  • question prompt and expected response form
  • hint level, scope, and reveal-policy result
  • attempt_feedback evidence-linked formative feedback
  • uncertainty reason, affected content, and next action
  • artifact_updated permitted file ID, type, and revision
  • session_state queued, thinking, waiting, paused, failed, or complete
  • Every action and event has a stable ID, session ID, actor, timestamp, schema version, and sequence number.
  • Action acknowledgement is persisted before agent execution so reconnect and retry are safe and idempotent.
  • Raw provider chunks may be converted into partial display text, but only validated complete events may change grading, progress, schedules, or files.
  • The UI renderer ignores privileged internal fields and never accepts an arbitrary component name, HTML fragment, shell command, or local path from the agent.

Context isolation and answer protection

  • Material visibility labels: every file is marked child-visible, tutor-usable, grader-only, or parent-only. The context broker selects by label and workflow, not by folder discovery alone.
  • Separate context packages: a tutor run never receives parent-private notes or a solution key unless the answer policy permits a narrowly scoped use. A grader may receive the answer key but its raw contents do not enter the child transcript.
  • Restricted filesystem root: the agent receives a session-specific directory containing copies or read-only links to approved files. It cannot traverse the family archive, AgentWorks data, home directory, or another child’s workspace.
  • Restricted tools: education workflows expose only named capabilities such as read permitted material, write session artifact, emit event, and request parent review. Shell, arbitrary network, and unrestricted local file tools are off by default.
  • Output validation: agent-created JSON or artifacts are schema checked, scanned for disallowed paths/content, and permission checked before indexing or rendering.
  • Reveal service: full-solution decisions pass through one server-owned policy function that records the request, attempts, hint levels, assignment mode, parent setting, decision, and reason. Prompt instructions alone are not sufficient.

  • Integration tests: SQLite transaction plus filesystem write, upload processing, tmux resume/recreate, stream reconnect, child switch, parent relock, scheduler, and WhatsApp routing.

Build order for minimum wasted effort

  1. Prove one vertical loop: a parent creates one child and subject, uploads one assignment, and starts a guided learning session.
  2. Add child mode: the child attempts the assignment through a restricted tutor interface.
  3. Add careful grading: submit a photograph, display transcription confidence, and route uncertain work to the parent.
  4. Add learning memory: save concept progress and use it to personalize the next session.
  5. Add schedules: turn assignment deadlines and learning gaps into a simple study plan.
  6. Add parent-only WhatsApp: capture assignments and deliver summaries after the app workflow is stable.
  7. Publish a separate pilot build: distribute the SparkQuill Electron installer privately to invited technical families without changing the AgentWorks experience.
Avoid false reuse

Reusing a large technical screen and hiding most of it is usually more work than creating a small purpose-built screen. Optimize for reuse of services, data models, hooks, renderers, and design primitives—not visual similarity to the current agent workspace.
