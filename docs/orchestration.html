MCP Bridge Wiring & Tool Exposure

How a CLI coding agent (Claude Code, etc.) discovers and calls tools through the MCP bridge — why it's fully dynamic even in the packaged DMG, and exactly why a registered tool can still be invisible to the agent.

01The big idea

CLI agents never get tools as native function-calls. They run in code-execution mode and reach every tool through an HTTP MCP bridge. So the real question is never "does the agent have tool X" — it's "is X published into the bridge for the exact agent instance this session is scoped to". That scoping is the whole story.

02What the DMG ships (static) vs what's dynamic

🖥️ Go agent server static binary

Holds the live, per-session, per-agent tool registry (a.customTools). Builds system prompts and answers get_api_spec + tool-execution HTTP calls.

🌉 mcpbridge static binary

In ~/go/bin/mcpbridge. A generic stdio↔HTTP proxy. Knows nothing about specific tools — forwards calls, scoped by env headers, to the Go server.

⌨️ CLI agent static binary

Claude Code / Gemini / etc. Launched as a subprocess; connects to mcpbridge over stdio as an MCP server named api-bridge.

Nothing about tools is baked into the DMG. The bridge and CLI are generic; the tool set is computed fresh, per session, by the Go server at runtime. A tool-registration fix ships by rebuilding the Go binary alone.

03Per-session wiring — BuildBridgeMCPConfig()

When a session uses a CLI provider, the Go server writes an MCP config telling the CLI to launch mcpbridge, passing everything through environment variables (mcpagent/agent/coding_agents_bridge.go):

The bridge (cmd/mcpbridge/main.go) does exactly this on every call:

get_api_spec is not a canned list in the bridge — every call round-trips to the Go server, which reads the current a.customTools of the scoped agent instance and returns it, grouped by category (category = the "server" name). The CLI's system prompt is built the same way (buildToolIndex + inlined pre-discovered specs) and rebuilt when the registry changes.

05How it stays in sync — the dynamic link

One shared structure connects the workflow code and the bridge: Agent.customTools. The workflow layer writes it; the bridge layer reads it.

// workflow layer (agent_go) registers →
RegisterCustomTool(name, def, params, exec, category)  →  a.customTools[name] = {def, category}
// bridge layer (mcpagent) reads, on demand and at prompt-build →
UpdateCodeExecutionRegistry()  →  buildToolIndex()              // the get_api_spec index
                               →  buildPreDiscoveredToolSpecs() // full specs inlined in the system prompt
                               →  rebuildSystemPromptWithUpdatedToolStructure()

For a tool to be callable by a CLI agent, all three must hold for the scoped agent:

1 · Registered

In that agent's a.customTools with a non-empty category.

2 · Allow-listed

GetToolsForWorkshopMode → SetToolAllowList; gates both the index and execution.

3 · Prompt rebuilt

UpdateCodeExecutionRegistry triggers it; SetToolAllowList alone does not.

06How this explains the bug

Symptom: a tool (notify_user) is registered in the session, yet the CLI agent can't see or call it; get_api_spec omits it from human_tools.

  1. The CLI sees one agent's registry. get_api_spec is tagged with X-Virtual-Scope-ID → the Go server returns the a.customTools of that one scoped instance (the conversational/workshop "parent" agent), not the union of all agents in the session.
  2. The tool registered on a different instance. Logs show Registered custom tool … notify … for the session — but on the child / step-execution agents (their default tool set is workspace_advanced:* + human_tools:*), not the scoped parent.
  3. So it's "registered" and "invisible" at once. The scope routing is exactly the mechanism that lets both be true: present on the children, absent from the parent the CLI is pinned to.
  4. The monitor is the clean proof. Its scoped agent registered zero human tools, so notify could never appear there regardless.
Root cause: notify_user is not in the a.customTools of the scoped (parent/conversational) agent that serves get_api_spec. The allow-list fix (gate 2) was necessary but not sufficient — the tool must be registered onto that parent agent (gate 1).

07The fix

Register the human notification tool onto the workshop/phase (parent) agent's tool set, so it lands in that agent's a.customTools → appears in get_api_spec → callable.

The parent agent's tools come from PreparePhaseAgentTools() (base_orchestrator_tools.go), which selects ["workspace_advanced:execute_shell_command", "human_tools:*"]. The fix ensures the phase-agent path actually carries notify_user through to registration on the parent and that it is allow-listed for workshop/run modes.

Verify with the diagnostic, not the agent: the temporary 🔬 [TOOLINDEX_DIAG] log prints the scoped agent's full a.customTools with agent_ptr, category, and allow-state — confirming notify_user is present on the instance that serves get_api_spec. Then in a fresh chat: get_api_spec should list it under human_tools, and a call should execute.

08Debug checklist — "agent can't call tool X"

  1. Never trust the agent's self-report. Have it run get_api_spec and show the category, or read the server log.
  2. Which instance? Match the agent_ptr / scope serving get_api_spec against where the tool was Registered custom tool. Mismatch → scope/instance bug (this case).
  3. Registered on the scoped agent? TOOLINDEX_DIAG / Built tool index for that instance.
  4. Allow-listed? Applied tool allow list / buildToolIndex blocked custom tools.
  5. Mis-routed call? server=<guess> … not found in tool-to-server mapping means the tool wasn't cleanly in the spec under its category, so the agent invented a server.
  6. Right binary running? Confirm the desktop app version and that the server restarted.
